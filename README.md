# ABD 即时通讯系统 (后端工程)

**高性能、高可用分布式即时通讯 (IM) 后端架构项目**

---

### 🌟 项目简介

本项目是一个基于 Go 语言开发的高性能即时通讯 (IM) 后端服务。系统的核心难点与发力点聚焦于解决 **海量用户的百万级并发长连接管理**，并通过工程化手段保障消息传输的 **绝对可靠与强有序（不丢包、不乱序、不重复）**。在整体架构上采用了分层隔离与解耦的设计，令其能够横向扩展以承受千万级的并发规模与极端峰值的吞吐量冲击，为现代即时通讯业务提供坚实的底层通讯设施。

---

### 🛠️ 核心技术栈

系统采用云原生微服务开发最成熟的技术生态：

* **核心语言**: Go (1.24, 采用最新 toolchain 构建)
* **网络与框架层**: `Gin` (RESTful HTTP 短连接 API), `Gorilla WebSocket` (高度重构适配的长链接网关)
* **数据存储**: `MySQL` + `GORM` (核心结构化数据持久化读写), `Redis/go-redis` (极速热点数据缓存)
* **消息中间件**: `Kafka (IBM/sarama)` (高并发上行消息异步中转、系统解耦、“削峰填谷”)
* **基石组件**: 基于 `Snowflake (雪花算法)` 的分布式发号器, `JWT` (鉴权), `Singleflight` (并发访问数据库的安全原语)
* **可观测性**: `Prometheus (client_golang)` (网关在线并发连接等核心 Metric 业务暴露)

---

### 🏛️ 三层解耦型核心架构

为打破单体架构在常驻长连接资源占用下的高并发瓶颈，此项目实施了经典的 **Gateway(接入网关) / Logic(业务逻辑) / Pusher(分发推送)** 三层物理隔离拆分设计：

1. **Gateway 长接入层 (WebSocket Server)**: 
   * 专注于处理客户端的常驻长连接，维护 `{UserID: Session}` 的连接路由拓扑。
   * 无状态、极低负载，支持随着客户端数量增加无缝横向弹性扩展 (依赖前置负载均衡)。
   * 对于上行写动作（如发送消息），网关仅将封包推入 Kafka 即可立即向客户端发送 ACK 确认，保持极轻量的长连接保活状态。
2. **Logic 短连接与基础数据层 (HTTP API)**: 
   * 管理低频、重 IO 的基础通讯状态，涵盖：用户鉴权、会话管理、拉取历史漫游聊天信息等功能。独立集群运行，其 IO 拥塞即使在极端情况下也绝不会影响核心 Gateway 层推送的稳定性。
3. **Pusher 消息异步路由引擎层**: 
   * 充当高吞吐量分发机。负责消费 Kafka 的中转队列，针对每条下行消息，高速去分布式 `Redis` 在线全局路由池解析目标设备的 `Node` 和 `Session`。
   * 精确将下行推送指令 RPC 回调给对应的 Gateway 节点，由其执行实际的 Socket Write 推送动作。

### 💡 核心技术特性与架构亮点

#### 📌 1. 极致的内存开销控制与 GC 调优实践 (池化与无锁化)
大量并发用户群频繁的上下线断链重置会导致海量的堆内存分配，从而引发严重的垃圾回收 (`GC`) 停顿延迟并最终卡断正常的长连接心跳。
* **Zero-Allocation 对象复用**: 针对整个生命周期最核心的 `Client` 连接对象结构体，底层全面引入 `sync.Pool` 资源池化技术。当 WebSocket `Upgrade` 握手建立时执行池 `Get` 获取复用指针，在 `UnRegister` 离线后重置关键字段并 `Put` 回池中。在常态流量下实现了海量长连接握手状态下的对象“零分配”，极大压低了 GC 毛刺与系统延迟等待时长。
* **无锁读写分离架构**: 为消除长连接并发读写资源竞争带来的锁阻塞损耗，底层采用原生 `Channel` 配合 Goroutine 内循环来执行“序列化读写”，深刻践行 **“Don't communicate by sharing memory; share memory by communicating”** 的并发准则。

#### 📌 2. 高可用缓存穿透防护墙设计
针对数百、甚至数千大群中热门话题导致的突发性并发读库（如：大量用户同时拉取群配置、群成员信息引起的流量波峰）。一旦 Redis 缓存过期或穿透失误，极易引发“雪崩效应”击垮后方 MySQL 存储。
* **分布式并发合并阻断**: 底层全面采纳 Go 高级同步原语 `"golang.org/x/sync/singleflight"` 进行方法级加固。
* **工作流**: 成功从内存维度拦截短时间内相同的超高并发查询。如果突发上百个针对同一个穿透 key 的 DB 携程，只会放行唯一的实体探针下探 MySQL 并随之回填 Redis。其余数百个被阻截的携程则阻塞等待该探针完毕，直接共享其内存返回结果。一劳永逸抵御了穿透/击穿造成的衍生雪崩灾难。

#### 📌 3. 消息全局高可靠保证与强有序同步模型（Seq Pull 机制）
避免 IM 中最致命的“看漏、发乱、重复收发”客诉问题，系统摒弃了单纯依靠网络层 ACK 以及无序 PUSH 的天真模型。
* **可靠持久化体系**: 上行消息入网关即双活推入 Kafka 提供高可用持久中转（保障不可逆与强制落库防丢）。
* **单调递增 Seq 同步差集防丢**: 通过独家 Snowflake 发号器结合 `Redis INCR` 为每一个会话 (Conversation) 分配绝对单调递增的 Seq 增量序号。
* **Pull 同步为王**: 客户端不但被动消费 Push 队列，并在本地维护当前已读的 `LastSeq`。当遇到弱网、网络切换重连时，客户端将其自身 `LastSeq` 和服务端的 `MaxSeq` 对比并主动发起 `Pull` 增量同步请求。从根本结构上彻底消除了发包无序、弱网漏包问题，实现客户端消费的完全“幂等性”。

#### 📌 4. 在线流量监控与容器可观测度量提取
由于 Gateway 网关层的核心连接数监控极其重要，系统已集成原生 Prometheus 暴露机制。
* 利用底层的 `atomic.Int64` 无锁原子计数器实时准确追踪并发注册池（在线/离线用户状态）。直接暴露指标仪表盘至 `/metrics` API (如暴露自定义变量 `OnlineUserGauge`)。为后续容器集群(K8S)实现 HPA(自动伸缩)提供高可用监控底座。

### 📂 后端工程目录布局 (Standard Go Layout)

```text
backend/
├── cmd/               # 应用入口二进制 Main/
├── config/            # YAML 系统配置文件支持
├── internal/
│   ├── api/           # HTTP API 控制器层
│   ├── im/            # WebSocket 核心通讯引擎层 (并发池化、网络层处理)
│   │   ├── client.go    # Client 连接单体抽象与双向流动管道
│   │   ├── ws_server.go # Main WebSocket Server 守护与池化持有区
│   │   ├── distributor/ # 全局消息流向调度处理区
│   │   └── pusher/      # 下行 Push 写扩散模块
│   ├── model/         # MySQL GORM 实体定义表
│   ├── service/       # 控制器调用核心逻辑层
│   └── pkg/           # 内部公共业务组件 (带依赖)
│       ├── database/  # 连接池初始化封装
│       ├── redis/     # 实现 Singleflight 加固的执行器
│       ├── kafka/     # 高可用 Pub/Sub 消费者组定义
│       └── prommetrics/ # Prometheus Exporter 数据埋点项定义
├── pkg/util/          # 零依赖工具集 (Jwt鉴权解析器、Snowflake等)
└── deployments/       # 运维部署设施编排配置
```

### 🚀 快速启动指南

**容器化一键编排启动 (本地完整环境验证)**：
```bash
# 包含并启动所有必须的伴生服务设施 (MySQL / Redis / Kafka) 以及后端主程序
cd deployments/build
docker-compose up -d

# 跟进容器运行状态与主节点日志
docker-compose logs -f backend
```

**单节点纯开发调试起步**：
```bash
# 确保 config/config.yaml 中的各个储存服务和中间件地址可用并提前创建好库
make run

# 运行工程集成单元测试模块
go test ./... -v
```

> **注意**: 有部分集成测试用例为强依赖外部测试节点环境（例如：指定联调网段内存在的 Kafka Server `192.168.6.130` 等），如在异构网络执行请首先修正对应的 IP 指向设定。
