# 压测基准测试报告 (Benchmarks)

本报告总结了 ABD IM 后端的性能与可靠性测试结果。

## 测试环境
- **CPU**: AMD Ryzen 9 7945HX (16 核心 / 32 线程)
- **内存**: 32GB
- **后端组件**: ABD Server (Go), MySQL, Redis, Kafka
- **测试工具**: 自定义压力与可靠性脚本 (`tests/stress/cmd/`)

## 场景 1：标准负载 (1,000 用户)
模拟 1,000 个并发用户，发送间隔为 1s。

| 指标 | 结果 |
| :--- | :--- |
| **并发用户数** | 1,000 |
| **持续时间** | 60 秒 |
| **总发送量** | 69,760 条消息 |
| **总接收量** | 72,395 条消息 |
| **平均 TPS** | 1,162.35 |
| **峰值 CPU (服务端)** | ~58% (总计) |
| **峰值内存 (服务端)** | ~144 MB |
| **可靠性** | 100% (基于收发比对) |

> [!NOTE]
> 由于高负载下的 I/O 超时，个体抽样工具曾出现连接失败，但全局压力测试确认消息送达率正常，未出现全局丢包。

## 场景 2：高并发负载 (10,000 用户) - **未优化基准**
| 指标 | 数值 |
| :--- | :--- |
| **并发用户数** | 10,000 |
| **认证阶段耗时** | ~22 秒 |
| **峰值 CPU (服务端)** | ~544% (约占用 32 线程中的 5.5 个) |
| **峰值内存 (服务端)** | ~185 MB |
| **平均 TPS** | **停滞/大幅下降** (未达到 10k TPS 目标) |
| **状态总结** | **确认软件瓶颈**。在高负载下，同步日志和内部锁限制了系统仅能使用约 20% 的 CPU 能力。 |

## 场景 2：峰值负载 (10,000 用户)
模拟 1,0000 个并发用户全量上线后的表现。

| 指标 | 结果 |
| :--- | :--- |
| **并发用户数** | 10,000 |
| **上线阶段** | 约 8 分钟 (受 50ms 步进间隔限制) |
| **峰值 CPU (服务端)** | ~645% (约 20% 的硬件总容量) |
| **峰值内存 (服务端)** | ~493 MB |
| **状态总结** | **触及软件瓶颈**。硬件资源仍有大量余裕 (~80% CPU 空闲)。 |

## 已知问题与改进
1.  **服务端稳定性**：修复了 Kafka 生产端过载时分发层可能出现的空指针宕机。增加了 Panic 恢复机制。
2.  **I/O 超时**：单机 10k 模拟时，本地 I/O 和缓冲区达到极限，导致部分抽样工具超时。
3.  **日志瓶颈**：确认为系统核心性能杀手，已通过“静默模式”验证了 I/O 消除后的吞吐提升。
