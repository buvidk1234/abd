# ABD IM 后端：第一次正式性能压测报告
**日期**: 2026-03-14
**测试目标**: 建立 10,000 用户量级的未优化性能基准 (Baseline)

## 1. 环境准备
测试前已执行 `clear_data.go` 脚本，确保环境处于以下纯净状态：
- **MySQL**: 所有业务表已 `TRUNCATE` 清空。
- **Redis**: 执行 `FLUSHALL` 完毕。
- **Kafka**: 话题 (`coming_message_topic`, `online_push_topic`) 已删除并重建。
- **服务端模式**: 开启 `ABD_SILENT=1`（静默模式，关闭日志输出）。

---

## 2. 硬件与环境规格
- **CPU**: AMD Ryzen 9 7945HX (16C/32T)
- **内存**: 32GB DDR5
- **存储**: NVMe SSD
- **后端**: Go 1.22+ ABD Server

---

## 3. 测试方案
- **并发用户**: 10,000 个 WebSocket 模拟。
- **步进间隔**: 10ms (连接建立阶段)。
- **发信频率**: 每个在线用户每秒发送 1 条消息 (1对1 随机目标)。
- **期望注入**: 10,000 TPS。

---

## 4. 实测指标 (基准数据)

| 指标项目 | 实测结果 | 备注 |
| :--- | :--- | :--- |
| **认证耗时 (Auth)** | ~20s | 并发 50 认证，速度极快 |
| **消息注入速率** | **10,000+ TPS** | 发信侧已达到预期压力 |
| **数据库处理速率 (Processed)** | **~600 - 800 TPS** | 确认系统最大入库能力 |
| **服务端 CPU 占用** | **~1100%** | 软件逻辑达到吞吐上限 |
| **服务端 内存占用** | ~380 MB | 内存表现极为优秀且稳定 |
| **磁盘写放大** | **约 4x** | 序列号、Timeline 等导致严重放大 |

---

## 5. 核心结论：为什么无法达到 1万 TPS？

经过本次“零干扰”基准测试，我们确认了系统当前的**三大软瓶颈**：

1.  **数据库入库能力 (MySQL Transaction)**：
    当前 MySQL 的物理吞吐量限制了最终的消息处理速度。即便 CPU 有大量余裕，单个消息的事务开销限制了 TPS 只能徘徊在 600-800 之间。

2.  **批处理失效 (Batcher Granularity)**：
    目前的 `BatchProcessor` 以 `ConversationID` 为键。在 10,000 人 1对1 场景下，Batcher 无法合并不同会话的消息，导致退化成了“单条插入”。

3.  **计算资源浪费**：
    Server 消耗了 1100% 的 CPU 但只产出了 800 TPS 的入库，说明在协程调度、序列号同步分配上存在较大的资源浪费。

---

## 6. 后续优化路线 (下一阶段)
- [ ] **全局批处理**：不按会话分组，直接进行全量全局的 SQL 合并插入。
- [ ] **异步序列号分配**：引入 Redis 号段预分配，彻底消除每条消息更新 `seq_conversations` 的事务。
- [ ] **Timeline 异步化**：进一步剥离从消息到多端同步记录的写入压力。

**本报告作为优化前的唯一 Baseline 存档。**
